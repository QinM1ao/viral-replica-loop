# Seedance 2.0 提示词复杂度与口播标签调研

日期：2026-07-16
范围：中文 R2V / 多参考图、约 13 秒、多镜头、原生音频口播的电商复刻
来源限制：只采用 ByteDance Seed、火山方舟 / BytePlus ModelArk 和 Runway 的第一方资料；访问日期均为 2026-07-16。

## 结论

当前提示词的**镜头时间轴值得保留，但声音标签写得过长**。`男主播画面内同期口播` 不是 Seedance 2.0 官方要求的固定术语；本轮检索到的官方资料也没有给出证据，证明把 `男主播说` 扩写成这类长标签能提高遵循率。

对本项目更合适的最小写法是：

- 人物在画面中说话：`男主播说：{台词}`；只有需要强调口型时再写 `男主播看镜头说：{台词}`。
- 人物不在画面中或明确不需要口型：`旁白：{台词}`。
- 无音乐：全局写一次 `无背景音乐`，不必在每个 Shot 重复。
- 音效：只写该镜头真正需要的动作音，如 `<指腹涂泥声>`。
- 台词落点：继续用 `时间｜Shot` 把台词放到对应画面旁边。比如产品名必须在产品特写出现时说，真正有效的控制是 `Shot 04 产品特写时，男主播说：{孔凤春清洁泥膜。}`，不是反复增加“画面内、同期、口播”等同义标签。

一句话概括：**删掉职业术语堆叠，保留“谁说、说什么、在哪个 Shot 说”这三个信息。**

## 官方证据

### 1. 官方明确建议简洁、避免重复

BytePlus / 火山方舟的 Seedance 2.0 官方提示词指南称，多模态素材可降低提示词书写门槛，并建议描述“尽可能简洁”、避免冗余和语义冲突。该指南确实给了更完整的高级公式，但并没有要求每条指令都把公式中的所有字段写满；复杂视频才需要按时间顺序拆镜头。[BytePlus ModelArk：Dreamina Seedance 2.0 series prompt guide](https://docs.byteplus.com/api/docs/ModelArk/2222480)；[火山方舟同源中文文档](https://www.volcengine.com/docs/82379/2222480?lang=zh)

Runway 的 Seedance 2.0 官方指南也建议使用正向、无歧义、以结果为中心的语言。它的示例很短，例如只写人物说一句话后跳出飞机，或只写“用 Image 1 的分镜指导场景”，没有附加固定的“同期口播”标签。[Runway：Creating with Seedance 2.0](https://help.runwayml.com/hc/en-us/articles/50488490233363-Creating-with-Seedance-2-0)

因此，“提示词可以很长”与“提示词应该尽量长”是两回事。BytePlus API 虽给出提示词建议上限为 1000 words，但这是输入边界，不是质量目标。[BytePlus ModelArk：Seedance 2.0 API Reference](https://docs.byteplus.com/en/docs/modelark/1520757)

### 2. 对话没有必须使用的长术语

官方材料同时使用多种自然写法：

- ByteDance Seed 官方发布示例直接写“所有人一起喊：……”；另一个示例写结尾出现 voiceover，没有使用“画面内同期口播”这类固定标签。[ByteDance Seed：Seedance 2.0 Official Launch](https://seed.bytedance.com/en/blog/seedance-2-0-official-launch)
- BytePlus 官方指南的对话示例使用“女孩微笑着说……”“男人说……”；旁白示例使用 `Voiceover: A deep, serene male voice says...`。这说明需要表达的是说话人、说话方式和台词，而不是某个强制关键词。[BytePlus ModelArk：Dreamina Seedance 2.0 series prompt guide](https://docs.byteplus.com/api/docs/ModelArk/2222480)
- Runway 官方示例使用 `the man ... says "..."`，同样没有把“说话”扩写成制作现场术语。[Runway：Creating with Seedance 2.0](https://help.runwayml.com/hc/en-us/articles/50488490233363-Creating-with-Seedance-2-0)

结论是：`男主播说` 足以表达画面内说话；`旁白` 足以表达画外发声。只有当人物是否张嘴存在歧义时，才补 `看镜头说` 或 `不出镜旁白`。

### 3. 官方更重视时间顺序与镜头内音画对应

BytePlus 官方指南将复杂视频的理想形式概括为按时间线组织的分镜：每个镜头依次写人物、地点、动作和运镜，并在对应镜头描述声音、人物声音和音乐。也就是说，音画对齐的主要手段是**把声音放进相应的镜头节点**，而不是把声音类型名称写得更长。[BytePlus ModelArk：Dreamina Seedance 2.0 series prompt guide](https://docs.byteplus.com/api/docs/ModelArk/2222480)

ByteDance Seed 官方发布也说明模型以自然语言接收多模态指令，支持 15 秒多镜头联合音视频生成，并强调对白、音效、音乐与画面节奏的联合生成。官方示例按事件顺序把人物说话、笑声和画面变化放在一起。[ByteDance Seed：Seedance 2.0 Official Launch](https://seed.bytedance.com/en/blog/seedance-2-0-official-launch)

所以对当前项目而言，`Shot 04 产品特写时说产品名` 比 `男主播画面内同期口播` 更有控制价值。

### 4. 大括号比长标签更接近官方结构化写法

BytePlus 官方指南建议用符号区分信息类型：对话用 `{}`、音效用 `<>`、音乐用 `（）`。这不是 API 的强制语法，但属于官方推荐的可读性辅助写法。[BytePlus ModelArk：Dreamina Seedance 2.0 series prompt guide](https://docs.byteplus.com/api/docs/ModelArk/2222480)

因此本项目可以把：

```text
声音：男主播画面内同期口播：“孔凤春清洁泥膜。”
```

简化为：

```text
男主播说：{孔凤春清洁泥膜。}
```

若需要绑定产品镜头，再写：

```text
Shot 04 产品正面特写时，男主播说：{孔凤春清洁泥膜。}
```

### 5. 没有官方证据证明长标签更好

本轮查阅的 ByteDance Seed 发布页、BytePlus / 火山方舟提示词指南与 API 文档、Runway Seedance 2.0 指南中，均未发现：

- 必须使用“画面内同期口播”或其英文等价词的要求；
- “同期口播”比“人物说”具有更高遵循率的对照实验；
- 标签越长，音画同步越好的结论。

因此，长标签带来收益的说法目前没有第一方依据。相反，官方资料明确支持自然语言和简洁、无歧义的描述。是否在本项目的具体网关和模型版本上存在微小差异，只能通过同素材 A/B 实测确认，不能从官方资料推断。

## 当前 Part 1 的问题

当前 [Part 1 提示词](../../output/job-011/seedance/seedance_part1_prompt.txt) 共约 1266 个字符，其中 `男主播画面内同期口播` 重复 5 次，`男主播画外音旁白` 出现 1 次。它没有超过官方输入上限，但存在三个工程问题：

1. 重复标签占用注意力，却没有增加新的镜头信息；
2. “画面内、同期、口播”三个词大部分语义重叠；
3. 真正重要的产品名落点只在 Shot 03–04 的括号里解释，容易被长句淹没。

## 适合本项目的最小可靠格式

全局只保留一次：

```text
约13秒，9:16真实手机短视频。严格按@图片1的Shot顺序和硬切节奏。
同一男主播、同一房间；无字幕，无背景音乐，只保留口播、环境声和动作音效。
```

单个时间块写：

```text
2.2–4.7秒｜Shot 03–04
画面：Shot 03 男主播把乳白厚泥涂到鼻翼和脸颊；硬切 Shot 04 产品正面特写。
声音：Shot 03 男主播说：{管它黑头还是油光。}；Shot 04 产品特写时说：{孔凤春清洁泥膜。}
音效：<轻微涂泥声、罐身移动声>
```

画外音则写：

```text
0.0–2.2秒｜Shot 01–02
画面：……
声音：旁白：{黑头闭口，涂！油皮粉刺，涂！}
音效：<轻微指腹触肤声>
```

这套格式保留了必要的音画控制，同时删除了没有官方依据的标签堆叠。

## 哪些是官方结论，哪些仍需实测

| 判断 | 证据等级 |
|---|---|
| Seedance 接受自然语言对话，`人物说：{台词}` 是官方示例支持的写法 | 官方直接支持 |
| 复杂视频适合按时间线 / Shot 组织，声音应靠近对应镜头 | 官方直接建议 |
| 描述应尽可能简洁、避免冗余和冲突 | 官方直接建议 |
| “画面内同期口播”是必需标签 | 官方不支持 |
| 删成长标签后，本项目音画落点一定更准 | 合理推断，仍需 A/B |
| `{}` 比中文引号在当前网关上一定更稳定 | 官方推荐格式，但增益大小仍需 A/B |

## 建议的 A/B 验证

如果要把结论升级为项目实证，只改一个变量：

- A 版：当前长标签格式；
- B 版：`男主播说：{}` / `旁白：{}` 的精简格式；
- 两版使用相同参考素材、模型、时长、分辨率和台词；每版至少生成 3 条，分别检查台词完整率、产品名是否落在 Shot 04、口型是否只发生在画面内说话镜头、是否误生字幕，以及镜头节奏。

在完成该 A/B 前，可以先把“精简标签”作为更符合官方指南的默认写法，但不要声称它已被项目实测证明更优。
